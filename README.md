# ПравоТруд

Минималистичный защищённый чат с полностью локальным RAG по ТК РФ. Генерация работает через Qwen2.5 1.5B Q4 (`llama.cpp`, CPU), эмбеддинги — через мультиязычный MiniLM в ONNX. Запросы не уходят во внешние AI API.

## Архитектура

Зависимости направлены внутрь: `presentation → application ← infrastructure`, а `application → domain`. Use cases зависят только от абстрактных портов, конкретные LLM, embedder, хранилище и reader подключаются в composition root `src/presentation/api.py`.

## Локальный запуск

```bash
cp .env.example .env
docker build -t pravotrud .
docker run --env-file .env -p 8000:8000 -v pravotrud-data:/data pravotrud
```

Откройте `http://localhost:8000`. Docker-образ уже содержит GGUF-модель; при первом индексировании или поисковом запросе отдельно загружается embedder.

## Загрузка ТК РФ

1. Добавьте актуальный PDF/TXT/MD в `documents/`.
2. Запустите сервис.
3. Перестройте индекс:

```bash
curl -X POST http://localhost:8000/api/index -H "X-Access-Key: change-me"
```

## Предварительная индексация перед деплоем

Чтобы Railway получил уже готовый индекс, положите ТК РФ в `documents/` и выполните локально:

```bash
python -m pip install fastembed pypdf
python -m scripts.index_documents
```

Команда создаст `artifacts/index/chunks.json`. Добавьте этот файл в Git вместе с проектом. При первом запуске контейнер автоматически скопирует готовый индекс в Railway Volume `/data`. Эмбеддинг-модель при обработке вопросов должна совпадать с моделью, использованной при индексации; поэтому не меняйте `EMBEDDING_MODEL` после сборки индекса.

После изменения embedding-модели или способа подготовки текста индекс обязательно нужно собрать заново. Текущая `paraphrase-multilingual-MiniLM` получает исходный русский текст без префиксов `query:` и `passage:`; эти префиксы относятся к семейству E5 и ухудшают качество данного индекса.

ТК РФ индексируется структурно по заголовкам `Статья N`. Статьи длиннее 2000 символов делятся с перекрытием 500 символов, а заголовок статьи повторяется в каждом продолжении. Это не позволяет RAG передать модели обрывок нормы без номера и названия статьи. Batch size предварительной индексации — 32.

`ENABLE_QUERY_REWRITE=false` оставляет исходный вопрос пользователя без LLM-переформулирования. Это рекомендуемый режим для маленькой локальной модели: embeddings + BM25 получают оригинальный текст, затем LLM-reranker выбирает финальные фрагменты из расширенного набора кандидатов.

Проверить готовый артефакт можно так:

```bash
ls -lh artifacts/index/chunks.json
```

## Railway

Создайте сервис из репозитория, добавьте Volume с mount path `/data` и переменную `ACCESS_KEY` со случайным длинным значением. Остальные переменные описаны в `.env.example`. Быстрый профиль для 8 CPU / 8 GB: `LLM_THREADS=8`, `LLM_CONTEXT=1536`, `LLM_BATCH=512`, `MAX_TOKENS=220`. `llama.cpp` собирается с OpenBLAS для ускоренной обработки RAG-промпта.

GGUF-модель и ONNX-эмбеддер загружаются во время Docker build и сохраняются внутри `/app/models`. Поэтому Railway не обращается к Hugging Face во время запуска или первого вопроса. Не задавайте для Railway старые пути `/data/...`: удалите переменные `MODEL_PATH` и `EMBEDDING_CACHE_DIR` либо установите их в `/app/models/model.gguf` и `/app/models/fastembed` соответственно.

Чтобы собрать образ с другой GGUF-моделью, передайте прямую ссылку как build argument `MODEL_URL`. Сам бинарный файл модели не хранится в Git: обычный GitHub не принимает файлы больше 100 MB, а Git LFS добавляет квоты и ещё одну точку отказа.

Температура генерации задаётся в Railway переменной `LLM_TEMPERATURE`. Допустимый диапазон — от `0.0` до `2.0`. Для юридического RAG рекомендуется `0.1–0.2`; значение по умолчанию — `0.15`.

Важно: статический ключ защищает от случайного публичного использования, но не является полноценной системой пользователей. Сайт работает только по HTTPS Railway; ключ хранится в localStorage браузера.
